## 什么是虚拟列表？

虚拟列表（Virtual List）是一种 **只渲染可视区域内 DOM 节点** 的优化技术。当你的列表有成千上万条数据时，全部渲染到 DOM 中会导致严重的性能问题。

### 核心思路

1. 监听容器的滚动事件
2. 根据滚动位置计算可视区域的起止索引
3. 只渲染可视区域内的列表项
4. 使用 `transform` 模拟完整列表高度

### 核心代码

```js
// 核心计算逻辑
const startIndex = Math.max(0, Math.floor(scrollTop / itemHeight) - buffer)
const endIndex = Math.min(
  startIndex + Math.ceil(containerHeight / itemHeight) + buffer * 2,
  totalItems
)

// 只渲染 startIndex ~ endIndex 范围内的数据
const visibleItems = allData.slice(startIndex, endIndex)

// 通过 transform 偏移，模拟滚动到正确位置
const offsetY = startIndex * itemHeight
```

## 📐 进阶版：动态高度

支持每个列表项具有不同的高度。核心难点在于：**渲染前无法知道真实高度**。

解决方案：
1. 使用「**预估高度**」初始化所有列表项的位置
2. 渲染后通过 **ResizeObserver** 记录每个列表项的真实高度
3. 使用缓存数组存储 `top/bottom/height`，通过**二分查找**快速定位可见范围

### 关键实现

```js
// 位置缓存结构
interface PositionItem {
  index: number
  top: number      // 元素顶部距离列表顶部的像素距离
  bottom: number   // 元素底部距离列表顶部的像素距离
  height: number   // 元素高度（初始为预估值，渲染后更新为真实值）
}

// 二分查找：给定 scrollTop，找到第一个可见的列表项索引
function binarySearch(scrollTop: number): number {
  let low = 0, high = positions.length - 1, result = 0
  while (low <= high) {
    const mid = Math.floor((low + high) / 2)
    if (positions[mid].bottom <= scrollTop) {
      low = mid + 1
    } else if (positions[mid].top > scrollTop) {
      high = mid - 1
      result = mid
    } else {
      return mid  // scrollTop 在 mid 的范围内
    }
  }
  return result
}

// 渲染后通过 ResizeObserver 更新真实高度
const observer = new ResizeObserver(() => {
  // 获取真实高度，更新缓存数组
  // 并级联更新后续所有项的 top/bottom
})
```

## ⚡ 最终版：动态高度 + 白屏优化

将动态高度和白屏优化合二为一的最终版本。支持通过开关自由切换动态/固定高度模式，并引入五大核心技术：

| 核心技术 | 作用 | 效果 |
|---------|------|------|
| **动态高度** | 预估高度 + ResizeObserver + 二分查找 | 每项可有不同高度，自动校准 |
| **requestAnimationFrame** | 节流滚动回调 | 多次 scroll 事件合并为每帧一次更新 |
| **IntersectionObserver** | 哨兵元素监测 | 减少对 scroll 事件的依赖 |
| **Web Worker** | 后台处理数据 | 排序/过滤不阻塞主线程 |
| **CSS contain** | 渲染隔离 | 浏览器只重绘变化的区域 |

### 优化详解

#### 1. rAF 节流

```js
// 多次 scroll 事件 → 只在下一帧更新一次
function onScroll(e) {
  pendingScrollTop = e.target.scrollTop
  if (rafId === null) {
    rafId = requestAnimationFrame(() => {
      scrollTop.value = pendingScrollTop // 只执行一次
      rafId = null
    })
  }
}
```

#### 2. CSS contain

```css
/* 告诉浏览器：每个列表项的渲染是独立的 */
.list-item {
  contain: layout style paint;     /* 布局/样式/绘制 隔离 */
  content-visibility: auto;        /* 不可见时跳过渲染 */
  contain-intrinsic-size: auto 40px; /* 预留占位尺寸 */
}
```

#### 3. Web Worker

```js
// 主线程：发送任务
worker.postMessage({
  type: 'sort',
  payload: { data, sortBy: 'value', sortOrder: 'desc' }
})

// Worker 线程：处理计算并返回
self.addEventListener('message', (e) => {
  const sorted = heavySort(e.data.payload)
  self.postMessage({ type: 'sorted', data: sorted })
})
```

## 关键知识点

### 为什么不直接用 `v-for` 渲染全部？

| 方式 | 10,000 条 | 100,000 条 |
|------|-----------|------------|
| 全量渲染 | ~3秒 白屏 | 浏览器卡死 |
| 虚拟列表 | ~20ms | ~20ms |

### 缓冲区（Buffer）的作用

在可视范围上下多渲染几行，减少快速滚动时的白屏闪烁：

```js
const buffer = 5
const startIndex = Math.floor(scrollTop / itemHeight) - buffer
```

### 白屏的根本原因

快速滚动时 scroll 事件过于频繁，JavaScript 执行与浏览器渲染抢占主线程：

1. scroll 事件高频触发（每秒可达数百次）
2. 每次触发都重新计算并更新 DOM
3. 浏览器来不及绘制新内容 → 出现白屏

**优化核心思路**：减少不必要的计算和渲染，把耗时操作移出主线程。
